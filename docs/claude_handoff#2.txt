RV32I QUICK REFERENCE
----------------------
R-type: funct7[6:0] | rs2[4:0] | rs1[4:0] | funct3[2:0] | rd[4:0] | opcode[6:0]
I-type: imm[11:0]   | rs1[4:0] | funct3[2:0] | rd[4:0] | opcode[6:0]
S-type: imm[11:5]   | rs2[4:0] | rs1[4:0] | funct3[2:0] | imm[4:0] | opcode[6:0]
B-type: imm[12|10:5]| rs2[4:0] | rs1[4:0] | funct3[2:0] | imm[4:1|11] | opcode[6:0]

CURRENT CPU ARCHITECTURE
------------------------
Pipeline stages:
  IF/ID (combined)  — riscv_fetch_decode.sv — combinational fetch and decode
  ID/EX register    — riscv_cpu.sv — latches decode outputs, flushes on branch
  EX                — riscv_execute.sv — register file, forwarding muxes, ALU
  EX/WB register    — riscv_cpu.sv — latches ALU result
  WB                — riscv_regfile.sv write port — synchronous register write

Forwarding unit (riscv_cpu.sv):
  fwd_a = ex_wb_reg_wr_en && (id_ex_rs1_addr == ex_wb_rd_addr) && (ex_wb_rd_addr != 0)
  fwd_b = ex_wb_reg_wr_en && (id_ex_rs2_addr == ex_wb_rd_addr) && (ex_wb_rd_addr != 0)

Forwarding muxes (riscv_execute.sv):
  assign fwd_rs1_data = fwd_a ? wb_data : rs1_data;
  assign fwd_rs2_data = fwd_b ? wb_data : rs2_data;
  assign rs2_imm      = alu_src ? imm : fwd_rs2_data;
  ALU input a = fwd_rs1_data
  ALU input b = rs2_imm

Branch flushing:
  Two-bubble penalty due to combinational fetch architecture.
  pc_src uses pipelined id_ex_branch and id_ex_func3:
    assign pc_src = id_ex_branch && ((id_ex_func3 == 3'b000 && zero) ||
                                      (id_ex_func3 == 3'b001 && !zero));
    pc <= pc_src ? (id_ex_pc - 32'd4 + id_ex_imm) : (pc + 32'd4);
  branch_stall = one-cycle delayed pc_src
  On pc_src: flush ID/EX (kill speculative instruction)
  On branch_stall: insert second bubble, hold PC

Register file writeback:
  Write address: ex_wb_rd_addr (passed as wb_rd_addr to riscv_execute)
  Write enable:  ex_wb_reg_wr_en (passed as wb_reg_wr_en to riscv_execute)
  Write data:    wb_data = ex_wb_mem_to_reg ? mem_data : ex_wb_alu_result
  Read is purely combinational, write is synchronous

WEEK 13 — RV32M MULTIPLIER
---------------------------
Architecture decisions:
  - Multiplier sits alongside ALU in EX stage
  - Main pipeline stalls (inserts NOPs into ID/EX) while MUL is computing
  - Handshake interface: start (1-cycle pulse in), done (1-cycle pulse out)
  - Pipeline only samples result when done asserts
  - Iterative shift-and-add, 32 cycles latency
  - Full 64-bit result internally
  - op selects: 3'b000=MUL (lower 32), 3'b001=MULH (upper 32)
  - busy flag prevents re-entry during computation

riscv_mul.sv — CURRENT CODE:
-----------------------------
module riscv_mul(
    input  logic        clk,
    input  logic        rst,
    input  logic        start,
    input  logic [31:0] op_a,
    input  logic [31:0] op_b,
    input  logic [2:0]  op,
    output logic [31:0] result,
    output logic        done
);
    logic [5:0]  bit_idx;
    logic [31:0] op_a_latched, op_b_latched;
    logic [63:0] op_a_shifted;
    logic [63:0] running_total;
    logic        busy;

    always_comb begin
        case (op)
            3'b000:  result = running_total[31:0];
            3'b001:  result = running_total[63:32];
            default: result = 32'b0;
        endcase
    end

    always_ff @(posedge clk) begin
        done <= 1'b0;
        if (rst) begin
            done          <= 1'b0;
            busy          <= 1'b0;
            bit_idx       <= 6'b0;
            running_total <= 64'b0;
            op_a_latched  <= 32'b0;
            op_b_latched  <= 32'b0;
            op_a_shifted  <= 64'b0;
        end else if (start) begin
            busy          <= 1'b1;
            op_a_latched  <= op_a;
            op_b_latched  <= op_b;
            bit_idx       <= 6'b0;
            running_total <= 64'b0;
        end else if (busy && (bit_idx != 6'd32)) begin
            bit_idx      <= bit_idx + 1;
            op_a_shifted <= 64'(op_a_latched) << bit_idx;
            running_total <= op_b_latched[bit_idx[4:0]] ?
                            (running_total + op_a_shifted) : running_total;
        end else if (busy && (bit_idx == 6'd32)) begin
            done <= 1'b1;
            busy <= 1'b0;
        end
    end
endmodule

SUSPECTED ROOT CAUSE
---------------------
The shift-and-add algorithm has a one-cycle pipeline delay between
op_a_shifted being computed and being used in the accumulation.

On cycle N:
  op_a_shifted <= op_a_latched << bit_idx   (shift by current bit_idx)
  running_total uses op_a_shifted from PREVIOUS cycle (cycle N-1)
  bit_idx <= bit_idx + 1

This means:
  - Cycle 0 (bit_idx=0): op_a_shifted gets shift-by-0, running_total uses
    reset value of op_a_shifted=0 — first bit always missed
  - Cycle 1 (bit_idx=1): op_a_shifted gets shift-by-1, running_total uses
    shift-by-0 result — correct for bit 0 but one cycle late
  - The result is that the accumulation is always one step behind

Additionally op_b_latched[bit_idx] is indexing with the CURRENT bit_idx
(before increment) — this may or may not be correct depending on the
intended algorithm.
